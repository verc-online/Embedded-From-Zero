# Struct и Enum

## История

Когда я только начинал писать программы на C, почти все данные представляли собой отдельные переменные.

```c
uint8_t hour;
uint8_t minute;
uint8_t second;
```

Пока программа небольшая — всё выглядит нормально.

Но спустя несколько файлов начинаешь замечать странную вещь.

Каждая функция принимает три аргумента.

```c
Display_ShowTime(hour, minute, second);
```

Через неделю появляется дата.

```c
Display_ShowDate(day, month, year);
```

Потом будильник.

Потом расписание кормления.

Потом состояние устройства.

И внезапно понимаешь, что вокруг уже не переменные, а объекты реального мира.

Есть время.

Есть дата.

Есть светодиод.

Есть кнопка.

Есть сервопривод.

Есть часы реального времени.

И каждый из них состоит сразу из нескольких связанных значений.

Именно в этот момент язык C предлагает два очень важных инструмента:

* `struct`
* `enum`

Именно они делают код похожим не на набор переменных, а на описание настоящего устройства.

---

# Постановка задачи

Представим обычные часы.

У них есть

* часы;
* минуты;
* секунды.

Можно хранить всё отдельно.

```c
uint8_t hour;
uint8_t minute;
uint8_t second;
```

Но возникает вопрос.

Что объединяет эти три переменные?

Ответ очевиден.

Все они описывают одно время.

Значит логично объединить их в одну сущность.

---

# Что такое struct

`struct` позволяет объединить несколько связанных переменных в один тип данных.

То есть сказать компилятору:

> "Эти значения всегда принадлежат одному объекту."

Например:

```c
typedef struct
{
    uint8_t hour;
    uint8_t minute;
    uint8_t second;
} Time;
```

Теперь вместо трёх отдельных переменных можно написать

```c
Time currentTime;
```

И обращаться так

```c
currentTime.hour
currentTime.minute
currentTime.second
```

Код сразу становится понятнее.

---

# Аналогия

Представьте папку с документами.

Без `struct` все листы лежат отдельно.

```
Часы

Минуты

Секунды
```

Со `struct` они лежат внутри одной папки.

```
Time
│
├── hour
├── minute
└── second
```

Папку можно передавать целиком.

Можно хранить несколько папок.

Можно создавать массив таких папок.

---

# Что происходит внутри памяти

Важно понимать одну вещь.

`struct` ничего магического не делает.

В памяти всё равно лежат обычные байты.

Например

```c
typedef struct
{
    uint8_t hour;
    uint8_t minute;
    uint8_t second;
} Time;
```

В памяти это выглядит примерно так.

```
Адрес

1000    hour
1001    minute
1002    second
```

То есть структура — это просто несколько переменных, расположенных рядом.

Именно поэтому указатель на структуру — это всего лишь адрес её первого байта.

---

# Обращение к полям

Есть два способа обращения.

Если переменная обычная

```c
Time time;

time.hour = 12;
```

Если указатель

```c
Time *time;

time->hour = 12;
```

Запись

```c
time->hour
```

означает

```c
(*time).hour
```

То есть

1. перейти по адресу;
2. получить структуру;
3. обратиться к её полю.

Именно поэтому в драйверах Embedded постоянно встречается оператор `->`.

---

# Где struct используется в Embedded

Практически везде.

Например состояние светодиода.

```text
Led
│
├── port
├── pin
└── state
```

Или кнопка.

```text
Button
│
├── port
├── pin
├── lastState
└── debounceTime
```

Или сервопривод.

```text
Servo
│
├── angle
├── minPulse
└── maxPulse
```

Или часы DS3231.

```text
RtcTime
│
├── hour
├── minute
└── second
```

Чем больше становится проект, тем больше данных начинают жить внутри структур.

---

# Что такое enum

Теперь другая проблема.

Допустим есть состояние устройства.

Можно написать

```c
uint8_t state = 0;
```

Но что означает ноль?

Никто не знает.

Через неделю уже и автор программы не вспомнит.

Поэтому используют `enum`.

---

# Пример

```c
typedef enum
{
    FEEDER_IDLE,
    FEEDER_WAITING
} FeederState;
```

Теперь код выглядит так.

```c
FeederState state;

state = FEEDER_IDLE;
```

Даже без комментариев понятно, что происходит.

---

# Что хранится внутри

Многие думают, что `enum` хранит строки.

Нет.

Компилятор заменяет имена обычными числами.

Например

```c
typedef enum
{
    RED,
    GREEN,
    BLUE
} Color;
```

На самом деле получится примерно так.

```
RED     = 0
GREEN   = 1
BLUE    = 2
```

То есть в памяти лежит обычное число.

Названия существуют только для программиста.

---

# Почему это важно

Сравните два варианта.

Без `enum`

```c
if (state == 1)
```

И с `enum`

```c
if (state == FEEDER_WAITING)
```

Во втором случае код можно читать почти как обычный текст.

---

# Struct и Enum вместе

Очень часто они используются одновременно.

Например.

```c
typedef enum
{
    LED_OFF,
    LED_ON
} LedState;

typedef struct
{
    volatile uint8_t *port;
    uint8_t pinMask;
    LedState state;
} Led;
```

Получается объект, который полностью описывает светодиод.

---

# Типичные ошибки

### Использование множества отдельных переменных

Вместо

```c
hour
minute
second
```

лучше создать одну структуру.

---

### Использование "магических чисел"

Плохо

```c
state = 2;
```

Хорошо

```c
state = FEEDER_WAITING;
```

---

### Создание слишком больших структур

Иногда новички пытаются сложить всё устройство в одну огромную структуру.

Лучше создавать небольшие логически законченные объекты.

---

### Использование `enum` как набора случайных чисел

Если перечисление описывает состояния устройства, оно должно содержать только состояния устройства.

Не стоит смешивать разные сущности в одном `enum`.

---

# Практическое применение

Практически каждый драйвер, который мы будем писать дальше, будет содержать структуры и перечисления.

Например:

* GPIO;
* UART;
* SPI;
* I2C;
* RTC;
* LCD1602;
* Scheduler.

По мере роста проекта именно они позволяют сохранить код понятным и расширяемым.

---

# Что должен уметь инженер

После этой главы инженер должен понимать:

* зачем нужен `struct`;
* почему связанные данные объединяют в структуры;
* как устроена структура в памяти;
* чем отличается обращение через `.` и `->`;
* зачем нужен `enum`;
* почему `enum` делает код читаемее;
* почему структуры и перечисления используются практически в каждом Embedded-проекте.

---

# Практическое задание

Не используя готовые библиотеки, самостоятельно спроектируйте:

1. структуру для кнопки;
2. структуру для сервопривода;
3. структуру для светодиода;
4. `enum` состояний кормушки;
5. `enum` состояний кнопки.

Пока не пишите функции. Подумайте только над тем, какие данные действительно должны храниться внутри каждого объекта.
