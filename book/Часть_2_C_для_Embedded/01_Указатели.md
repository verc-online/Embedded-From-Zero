# 01. Указатели

## История

Когда я только начал изучать язык C, именно указатели казались мне самой страшной темой. В интернете их называли сложными, в учебниках рисовали бесконечные стрелочки, рассказывали про двойные указатели, арифметику указателей и ещё десятки новых терминов.

Но больше всего меня смущал один вопрос.

**Зачем вообще это нужно?**

Я хотел научиться программировать микроконтроллеры, а не писать очередную программу, которая складывает два числа. Зачем мне знать какие-то адреса памяти?

Ответ пришёл только тогда, когда я впервые зажёг светодиод на микроконтроллере.

Оказалось, что практически весь Embedded построен именно на указателях.

Не потому, что так любят разработчики языка C.

А потому, что микроконтроллер устроен именно так.

---

# Постановка задачи

Представьте, что мы написали обычный код.

```c
int value = 10;
```

В памяти компьютера или микроконтроллера появилась переменная, в которой хранится число 10.

Теперь другой пример.

```c
PORTB = 0x01;
```

После выполнения этой строки на выводе PB0 появляется напряжение, и светодиод загорается.

Но почему?

Ведь мы тоже просто что-то записали.

Почему в одном случае изменилось число в памяти, а в другом — произошло физическое действие?

Ответ на этот вопрос и объясняет, зачем нужны указатели.

---

# Что такое указатель

Каждая переменная хранится где-то в памяти.

У памяти есть адрес.

Например:

```text
Адрес      Значение

0x0100  →     10
```

Переменная `value` хранится по адресу `0x0100`.

Обычно программисту не нужно знать этот адрес. Компилятор сам найдёт переменную по её имени.

Но иногда нам необходимо работать не с именем переменной, а с её адресом.

Именно для этого существуют указатели.

Указатель — это обычная переменная, которая хранит **не значение**, а **адрес другой области памяти**.

Например:

```c
int value = 10;
int *ptr = &value;
```

Здесь происходит следующее.

* `value` хранит число `10`.
* `&value` означает «получить адрес переменной».
* `ptr` хранит этот адрес.
* `*ptr` означает «обратиться по этому адресу».

Получается примерно такая картина.

```text
value
┌───────────┐
│     10    │
└───────────┘
      ▲
      │
   адрес 0x0100
      ▲
      │
ptr ──┘
```

Сам указатель не хранит число 10.

Он хранит только адрес, по которому это число находится.

---

# Разыменование

Когда мы пишем

```c
*ptr = 25;
```

происходит следующее.

1. Процессор берёт адрес, который хранится в `ptr`.
2. Переходит по этому адресу.
3. Записывает туда новое значение.

После этого переменная `value` тоже станет равной 25.

Потому что указатель не хранит копию переменной.

Он указывает на её настоящее место в памяти.

---

# Что происходит внутри микроконтроллера

До этого момента всё выглядело как обычное программирование.

Но теперь начинается Embedded.

Представим, что внутри микроконтроллера существует регистр PORTB.

У него тоже есть адрес.

Например:

```text
0x25
```

Этот адрес не хранит обычную переменную.

За ним скрывается настоящий аппаратный блок микроконтроллера.

Когда программа записывает значение по этому адресу, изменяется не просто память.

Изменяется работа электронной схемы.

Происходит примерно следующее.

```text
Код
        │
        ▼
PORTB = 0x01
        │
        ▼
Запись по адресу 0x25
        │
        ▼
Регистр PORTB
        │
        ▼
Логика порта B
        │
        ▼
Транзистор выхода PB0
        │
        ▼
На выводе появляется напряжение
        │
        ▼
Светодиод загорается
```

Вот почему Embedded так сильно отличается от обычного программирования.

Здесь запись в память может изменить состояние реального устройства.

---

# Почему это работает

Когда мы пишем

```c
PORTB |= (1 << PB0);
```

кажется, будто мы обращаемся к какой-то особенной переменной.

На самом деле это не так.

Во время компиляции имя `PORTB` превращается в обращение к конкретному адресу памяти.

Примерно так:

```c
*(volatile uint8_t *)0x25 |= (1 << PB0);
```

Для процессора уже не существует имени `PORTB`.

Он получает команду:

> «Запиши новое значение по адресу 0x25.»

Именно поэтому указатели являются фундаментом всей Embedded-разработки.

---

# Проектирование драйвера

Можно написать так:

```c
*(volatile uint8_t *)0x25 = 0x01;
```

Работать это будет.

Но представьте, что адрес регистра изменился.

Тогда придётся искать его по всему проекту.

Гораздо удобнее один раз сохранить адрес в указателе:

```c
volatile uint8_t *port = (volatile uint8_t *)0x25;

*port = 0x01;
```

Теперь, если адрес изменится, достаточно исправить только одну строку.

Именно поэтому хорошие драйверы стараются не разбрасывать «магические» адреса по всему коду.

---

# Типичные ошибки

**Неинициализированный указатель**

```c
int *ptr;
*ptr = 10;
```

Указатель содержит случайный адрес.

Запись по такому адресу может испортить память программы или привести к аварийному завершению.

---

**Путаница между `*` и `&`**

Очень легко перепутать две операции.

`&` получает адрес.

`*` обращается по адресу.

Запомнить можно простой фразой.

> Амперсанд (`&`) спрашивает: «Где находится переменная?»

> Звёздочка (`*`) говорит: «Покажи, что лежит по этому адресу.»

---

**Мыслить указателем как числом**

Указатель — это не значение.

Это путь к значению.

Почти всегда нас интересует не сам адрес, а данные, которые находятся по нему.

---

# Практическое применение

После этой главы становится понятно, почему практически каждый драйвер содержит указатели.

GPIO обращается к регистрам портов.

UART — к регистрам передачи и приёма данных.

SPI — к своим регистрам.

Таймеры — к своим.

Даже внешние устройства, подключённые по I²C или SPI, в конечном итоге управляются через регистры микроконтроллера.

Получается, что почти вся Embedded-разработка — это работа с памятью по заранее известным адресам.

---

# Что должен уметь инженер

После изучения этой главы вы должны понимать:

* зачем нужны указатели;
* чем указатель отличается от обычной переменной;
* что означают `*` и `&`;
* почему изменение значения по адресу меняет исходную переменную;
* почему регистры микроконтроллера доступны через указатели;
* каким образом запись в память приводит к изменению состояния реального оборудования.
